# Разработка Динамичной SvelteKit Игры (Памятка)

Этот проект — динамичная RPG "нулевого игрока" (Idle RPG). Большинство событий происходит на сервере автономно (через **Worker**), а клиент только наблюдает за этим (через **WebSocket**), изредка вмешиваясь.

При разработке следует помнить о важных архитектурных концепциях проекта и правилах работы:

## 1. Реактивность Svelte 5 и WebSocket
- **Проблема**: Компилятор Svelte 5 может предупреждать вас о том, что передача данных из `data` в руны `$state` (например, `let events = $state(data.offlineEvents)`) разрывает связь. Из-за этого может появиться желание использовать `$effect`, чтобы "синхронизировать" данные.
- **Правило**: **Используйте `$effect` только для жесткой навигации, не для реалтайм-данных!** Если вебсокет на клиенте сам пушит новые события в массив `events`, то хук `$effect(() => { events = data.events })` может внезапно перезатереть свежие пришедшие данные старыми из первоначального HTTP-запроса `data`.
- Вместо этого, создавайте локальные копии состояний на старте: `let initial = data.events; let events = $state(initial);`, и обновляйте `events` только через вебсокет или явные действия пользователя.

## 2. Изоляция Статических Данных в Worker
- **Проблема**: BullMQ `Worker` запускается как независимый фоновый процесс Node.js. 
- **Правило**: При импорте статических JSON-объектов или массивов (`items.ts`, `enemies.ts`) в `worker.ts`, убедитесь, что вы используете правильные названия экспортированных переменных (например, `initialItems`, а не просто `items`). Иначе Worker упадет с ошибкой "Cannot convert undefined or null to object". Деструктуризация должна быть точной.

## 3. Взаимодействие Клиент → Сервер (Вмешательство)
- В игре реализована "божественная сила". Обычные действия пользователя (такие как "Благословить", "Покарать") требуют обращения к базе данных и пересчета логики.
- Любое вмешательство происходит по следующему флоу:
  1. `fetch('/api/divine/...')` отсылает REST запрос.
  2. SvelteKit Endpoint (`+server.ts`) обновляет персонажа и записывает это в БД.
  3. Клиент оптимистично обновляет интерфейс и инициирует синхронизацию `refreshOfflineEvents`.
  4. Вебсокет подхватывает новые изменения через `Redis.publish` и докачивает ивент.

## 4. Vanilla CSS вместо Tailwind
- Из-за выбранного стека, мы используем **Vanilla CSS**.
- **Правило**: Чтобы интерфейс выглядел современно и консистентно (подобно библиотеке shadcn/ui), используйте **CSS Variables** в корневом пространстве: `--bg-deep`, `--border-light`, `--primary`, `--gold-bright`. 
- Не пишите сотни мелких инлайн-стилей. Вместо этого составляйте компонентные классы (`.card`, `.btn-primary`) в `<style>` тегах компонентов с жесткой привязкой к CSS переменным.

## 5. Drizzle ORM и Миграции Схемы
- BetterAuth требует жесткой схемы в `users`.
- Если вы добавляете новое поле к сущностям, **всегда предоставляйте значение `default()`** для `notNull()` полей, иначе `npx drizzle-kit push` заблокирует миграцию на существующих данных.
